# Repository Guidelines

Whatever action you can do yourself, please do yourself; this includes starting apps and verification.

## Project Structure & Module Organization

This repository will host the Axion waitlist landing page, built with Next.js. Keep application code in `src/`:

- `src/app/` for routes, layouts, and page-level styles (App Router).
- `src/components/` for reusable UI components; group related pieces in feature folders such as `components/hero/`.
- `src/lib/` for utilities, typed constants, and API clients.
- `public/` for static assets such as logos, SVGs, and social-preview images.
- `src/**/*.test.tsx` or `tests/` for automated tests, mirroring the code under test.

Avoid placing reusable components directly in route folders. Keep page composition in `src/app/` and move behavior or visual primitives into components.

## Build, Test, and Development Commands

After `package.json` is introduced, use npm unless the project explicitly adopts another package manager:

- `npm install` - install dependencies.
- `npm run dev` - start the local development server.
- `npm run build` - create a production build; run before opening a pull request.
- `npm run start` - serve the production build locally.
- `npm run lint` - run the configured lint checks.
- `npm test` - run the test suite when it is configured.

Do not add alternative lockfiles. Commit the lockfile generated by the selected package manager.

## Coding Style & Naming Conventions

Use TypeScript, functional React components, and two-space indentation. Prefer named exports for shared components and utilities. Use `PascalCase` for components (`EpisodePreview.tsx`), `camelCase` for functions and variables, and `kebab-case` for route folders. Keep Tailwind or CSS class lists readable by grouping layout, spacing, typography, and visual styles. Run the repository formatter and linter before committing.

## Testing Guidelines

Add tests for interactive behavior, data transformations, and regressions. Name tests after observable behavior, for example `waitlist-form.test.tsx` or `"submits a valid email"`. Prefer user-facing assertions over implementation details. New visual-only sections do not require tests unless they add logic.

## Commit & Pull Request Guidelines

There is no established Git history yet. Use concise, imperative Conventional Commit-style messages, such as `feat: add waitlist signup form` or `fix: validate email address`. Keep commits scoped and avoid unrelated formatting changes.

Pull requests should explain the user-facing change, link the relevant issue when available, list verification performed, and include screenshots or a short recording for visual changes. Never commit API keys; place local values in `.env.local` and document required variables in `.env.example`.
